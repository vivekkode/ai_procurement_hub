## Project Overview

The AI Hub Procurement Engine helps procurement managers in the Monterrey, Mexico region
make optimal supplier decisions for fuel purchases. It ingests daily and weekly pricing
data from 4 suppliers, normalizes it to a common schema, calculates landed costs
(base price + freight + tolls), and ranks suppliers with a full math breakdown.

**Current sprint:** Data Ingestion Layer (Week 3 — Functional Core Logic)

**Dataset:** Synthetic oil pricing data for academic use
- 4 suppliers: Valero, Exxon, Marathon, Pemex
- 64,078 clean records
- Jan 2024 → Jan 2026
- Products: Regular (87 Oct), Premium (91 Oct), Diesel
- All prices normalized to MXN per Liter

## Data Ingestion Pipeline

### Valero — HTML Parser

**File:** `src/ingestion/parse_valero.py`
**Input:** `data/raw/valero/valero_YYYYMMDD.html`
**Library:** BeautifulSoup4 + lxml

Each daily HTML file contains multiple `div.terminal-card` blocks, one per terminal.
The parser loops all cards automatically — adding or removing terminals requires
no code changes.

**Key design decisions:**
- Date extracted from filename (`valero_YYYYMMDD.html`) not from the "Effective Since"
  column, which contains a timestamp that varies by time zone
- Product names normalized from verbose format e.g. `"91 Oct RDP 10.0 RVP"` → `"Premium"`
- `prev_price` captured in `valero_clean.csv` but excluded from combined file
  (no equivalent in other suppliers — will be re-derived from historical data in normalization layer)
- "Save / Print" button text and "Change" column (arrow symbols) ignored — no data value

**Output:** 16,849 rows | 8 terminals | 545 dates

```python
from src.ingestion.parse_valero import parse_valero_folder, validate_prices

df = parse_valero_folder("data/raw/valero/")
df = validate_prices(df)
```

---

### Exxon — Excel Parser

**File:** `src/ingestion/parse_exxon.py`
**Input:** `data/raw/exxon/exxon_YYYYMMDD.xlsx`
**Library:** openpyxl

Fixed structure: Row 1 = title with date, Row 2 = blank, Row 3 = column headers,
Rows 4–12 = data (3 terminals × 3 products), Row 13 = blank, Row 14 = disclaimer.

**Key design decisions:**
- **Resilience fix:** Parser reads columns by name keyword, not by position index.
  `find_col(col_map, "facturaci")` finds the invoice price column regardless of
  whether new columns are inserted by the supplier. Falls back to position index
  with a WARNING log if column name is not found.
- Date uses DD/MM/YYYY format (day-first) — different from Valero's MM/DD/YYYY
- `discount` and `ref_price` captured in `exxon_clean.csv` only — Exxon is the
  only supplier providing a discount column
- Sheet name `"Precios Exxon"` accessed via `wb.active` — sheet name not stored

**Output:** 4,905 rows | 3 terminals | 545 dates

```python
from src.ingestion.parse_exxon import parse_exxon_folder, validate_prices

df = parse_exxon_folder("data/raw/exxon/")
df = validate_prices(df)
```

### Marathon — TXT Email Parser

**File:** `src/ingestion/parse_marathon.py`
**Input:** `data/raw/marathon/marathon_YYYYMMDD_COUNTRY-IT-City.txt`
**Library:** Python `re` (regex)

Each file is a plain-text email covering one terminal on one date. Both the date
and terminal name are encoded in the filename.

**Key design decisions:**
- Parser uses **label matching**, not positional reading — looks for `"Invoice Price"`,
  `"Base Price"`, `"IEPS 2D"` etc. by name. Row reordering inside the email body
  has zero effect on output.
- Full tax breakdown captured in `marathon_clean.csv`:
  `base_price`, `ieps_2d`, `ieps_2h`, `ieps_2a`, `unit_price`, `iva`, `invoice_price`
- Tax columns excluded from `all_suppliers.csv` per Roberto's guidance (Mar 13 meeting):
  *"focus on final prices rather than individual tax breakdowns for initial modeling"*
- Tax data preserved for future use in explainability module and tax sensitivity analysis
- `Net Price` row (always identical to `Base Price`) not captured — redundant
- Email headers (De, Sent on, To, CC, Asunto), separator lines, and both
  legal disclaimers completely ignored

**Output:** 16,350 rows | 10 terminals | 545 dates

```python
from src.ingestion.parse_marathon import parse_marathon_folder, validate_prices

df = parse_marathon_folder("data/raw/marathon/")
df = validate_prices(df)
```

### Pemex — PDF Parser

**File:** `src/ingestion/parse_pemex.py`
**Input:** `data/raw/pemex/reportes_pdf/pemex_tar_YYYYMMDD_YYYYMMDD.pdf`
**Library:** pdfminer.six

Weekly PDFs cover a date range (e.g. Jan 1–5). Each PDF has 6 pages:
2 pages per product type (Magna/Regular, Premium, Diesel).

**CRITICAL — Unit conversion:**
Pemex prices are in **MXN per cubic meter (m³)**.
All prices divided by 1000 before storing as `price_mxn_per_l`.
Original MXN/m³ value preserved in `pemex_clean.csv` as `price_mxn_per_m3`.

**Key design decisions:**
- Pages split by form-feed character (`\f`), parsed independently
- Each page: all 38–40 region names listed first, then all prices in the same order.
  Regions and prices collected into separate lists then paired by index position.
- Spanish date range `"del 1 al 5 de enero de 2024"` parsed via `MONTHS_ES` dictionary
- Product type detection limited to first 200 characters of each page (header area only)
  to avoid false matches — `"PEMEX DIESEL"` checked before `"premium"` keyword
- Known header strings (`REGION`, `PEMEX MAGNA`, `PETROLEOS MEXICANOS` etc.)
  in a skip list to prevent pairing with prices
- `date_end` captured in `pemex_clean.csv` only — combined file uses `date_start`
  for consistency with daily suppliers
- Disclaimer text on each page detected by `"sinteticos"` keyword and skipped

**Output:** 25,974 rows | 78 regions | 111 weeks

```python
from src.ingestion.parse_pemex import parse_pemex_folder, validate_prices

df = parse_pemex_folder("data/raw/pemex/reportes_pdf/")
df = validate_prices(df)
```

## Output Schema

All parsers output to a common schema. The combined `all_suppliers.csv` contains
these 11 columns:

| Column | Type | Example | Notes |
|---|---|---|---|
| `date` | datetime | `2024-01-01` | Start of validity period |
| `supplier` | str | `Valero` | Valero / Exxon / Marathon / Pemex |
| `terminal_id` | str | `TMX000001` | Supplier-specific terminal code |
| `terminal_name` | str | `Altamira` | Human-readable terminal name |
| `state` | str | `TMS` | State/region code where available |
| `country` | str | `MX` | MX or US |
| `product_type` | str | `Regular` | Regular / Premium / Diesel |
| `price_mxn_per_l` | float | `20.987` | Final invoice price MXN/Liter |
| `contract_type` | str | `Sin Marca` | Contract type label |
| `source_file` | str | `valero_20240101.html` | Original filename for traceability |
| `price_flag` | bool | `False` | True if price outside 15–35 MXN/L |

## Price Guardrails

All parsers include a `validate_prices()` function that flags records where
`price_mxn_per_l` falls outside the expected range of **15.0 – 35.0 MXN/L**.

Flagged records are not removed — a `price_flag = True` column is added so the
engine layer can decide how to handle them. This implements the guardrail requirement
from the Mar 13 client meeting:
> *"Implementation of hardcoded logic to prevent anomalous recommendations
> (e.g., erroneous $0.01/liter prices)"*

## LLM Fallback (Planned — Week 5)

For supplier files that do not match any of the 4 known formats, a fallback parser
`src/ingestion/parse_llm.py` will be added in Week 5. It will:
1. Send the raw file content to Claude API
2. Prompt for structured JSON extraction (date, terminal, product, price)
3. Convert the JSON response to the same 11-column schema
4. Flow into `all_suppliers.csv` transparently

This follows the architecture decision from the Mar 13 meeting:
> *"Direct parsing for structured data. LLMs reserved for unstructured formats."*
